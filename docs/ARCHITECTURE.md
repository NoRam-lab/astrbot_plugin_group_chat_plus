### Web 面板认证文件结构（安全分离）

当前版本将认证数据拆分为**两个独立文件**，实现密码与 JWT 密钥的物理隔离：

| 文件 | 存储内容 | Web 端可读 | 说明 |
|------|---------|:----------:|------|
| `web_data/auth.json` | 密码哈希、salt、hash_version、password_changed 标志 | ❌ 不可读 | 密码相关数据 |
| `web_data/jwt_secret.json` | JWT 签名密钥、临时密码明文副本、重载标记 | ❌ 不可读 | JWT 与会话管理数据 |

> **为什么分离？** 如果攻击者获取了包含密码哈希的文件，不应同时获得 JWT 签名密钥——否则可以绕过密码验证直接伪造登录令牌。两个文件的独立存储使单一泄露的影响范围最小化。

#### 自动迁移机制

如果你是从旧版升级，且旧版的 `auth.json` 中同时包含密码和 JWT secret，系统会在启动时自动执行以下迁移：

1. 检测到 `auth.json` 中存在 `jwt_secret` 字段 → 将其提取并写入新建的 `jwt_secret.json`
2. 从 `auth.json` 中删除 `jwt_secret` 和 `_web_initiated_reload` 字段
3. 迁移完成后 `auth.json` 仅保留密码数据，`jwt_secret.json` 管理 JWT 相关数据
4. 整个过程无需用户手动操作，日志中会输出迁移确认信息

#### 旧版根目录 auth.json 兼容

如果你是从更早的版本升级，并且旧版本的密码文件保存在插件数据**根目录**下的 `auth.json`（而非 `web_data/` 子目录），请在升级前先将该文件移动到 `web_data/auth.json`。否则新版本会在 `web_data/` 下重新生成默认密码。

如果你发现当前插件数据目录中同时存在：
- 根目录 `auth.json`
- `web_data/auth.json`

那么根目录那个通常是旧版本遗留文件；在确认当前 Web 面板已经正常使用 `web_data/auth.json` 登录后，可以手动删除根目录旧文件。

### Web 面板密码哈希升级说明

当前版本起，Web 面板密码默认使用 `Argon2id` 内存硬化哈希，替代旧版 `PBKDF2-SHA256`，显著提升抗 GPU 暴力破解能力。

- 新生成密码、用户主动修改后的密码、以及重置后生成的默认随机密码，都会直接保存为 `Argon2id`
- 若现有 `web_data/auth.json` 仍为旧版 `PBKDF2-SHA256` 哈希，用户使用原密码首次登录成功后会自动透明迁移到 `Argon2id`
- 迁移过程无需用户手动改密，也不会影响原密码本身

`web_data/auth.json` 会额外保存 `hash_version` 字段，便于后续排查和兼容判断。

### Web 面板安全防护体系

当前版本实现了多层安全防护：

#### 1. Content-Security-Policy (CSP)

所有页面均强制启用严格 CSP，防止 XSS 攻击注入恶意脚本：

- **script-src**：使用 **nonce（一次性令牌）** 替代 `unsafe-inline`，每个响应生成唯一 nonce 并注入到 `<script>` 标签中
- **style-src**：保留 `unsafe-inline` 以支持动态主题切换等运行时样式操作
- **default-src**：设为 `'none'`，仅允许显式声明的资源来源
- **connect-src / form-action / frame-ancestors**：限制为 `'self'` 或禁止

> 这意味着即使攻击者能在页面中注入 HTML，也无法执行任意 JavaScript 代码。

#### 2. 会话令牌与多端登录

当前 Web 面板使用 **JWT + HttpOnly Cookie + 服务端会话表** 的组合方式管理登录态：

- 登录成功后，服务端创建独立 `sid / device_id` 会话记录，并通过 `HttpOnly Cookie` 下发登录态
- 每次登录都会生成新的独立会话，因此 **多设备 / 多浏览器 / 多次登录默认允许并存，不会互踢**
- 登录页访问到有效会话时，会直接跳转到面板页，无需重复输入密码
- `web_panel_ip_bind_check=true` 时，会把登录 IP 绑定到会话中；普通请求、面板访问和心跳都会校验当前 IP
- 心跳接口只更新最近活跃时间，不会延长 24 小时绝对过期时间
- 改密码、密码重置、非 Web 面板发起的服务端重启，会让旧会话整体失效并要求重新登录

#### 3. 会话心跳机制

前端会通过低频心跳主动检测会话是否仍然有效：

- 前台默认每 `300` 秒检测一次
- 后台默认每 `1200` 秒检测一次
- 页面切回前台、网络恢复时会立即补一次检测
- 网络异常时不会立即判定断联，而是进入缓冲重试期，按 `15s → 30s → 60s → 120s` 逐步退避重试
- 只有真正收到 `401 reason`（如过期、IP 变化、密码修改、服务端重启）时，才会统一清理会话并要求重新登录
- 同一浏览器多标签页只会选出一个 `Leader` 标签页负责发心跳，其余 `Follower` 标签页只监听广播，避免重复探测

#### 4. 敏感文件保护

Web 文件管理 API 对以下敏感文件实施访问控制（返回 403）：

| 受保护文件/模式 | 保护原因 |
|----------------|----------|
| `auth.json` | 包含密码哈希 |
| `jwt_secret.json` | 包含 JWT 签名密钥 |
| `sessions.json` | 包含服务端 Web 会话表 |
| `access_log*` | 访问日志可能暴露 IP 信息 |
| `bans.json` | 封禁策略不应被篡改 |

这些文件在 Web 面板的文件浏览器中显示为"受保护"，无法查看或编辑内容。Web 会话管理中的聊天记录查看/编辑针对的是插件自定义 `chat_history/...` 历史文件，而不是这些核心安全文件。

#### 5. Web 会话管理、重置与图片缓存边界

当前 Web 面板中有三类容易混淆的动作：

- **会话管理中的单会话重置**
  - 仅清理该会话相关运行态
  - 删除该会话对应的自定义聊天记录文件 `chat_history/...`
  - 为该会话设置 `history_cutoff.json` 历史截止时间戳
  - 不应扩散到其他会话

- **命令面板中的全局重置**
  - 清理插件维护的全局运行态与相关本地持久化缓存
  - 删除自定义 `chat_history` 历史目录
  - 删除注意力/主动对话等长期持久化文件
  - 为所有已知会话设置历史截止时间戳
  - 不直接删除 AstrBot 官方 ConversationManager 对话历史

- **图片缓存清理**
  - 只清理图片描述缓存
  - 当前主缓存文件为 `image_cache/descriptions.jsonl`
  - 若检测到旧版残留路径 `image_description_cache.json`，清理逻辑也会兼容处理
  - 不清理聊天记录、注意力、概率或主动对话状态

#### 3. 默认密码日志输出机制

首次安装或密码重置后，系统会以 **WARNING 级别**向 AstrBot 日志输出默认密码及安全警告：

```
[WARNING] 🔑 Web 面板初始密码已随机生成: Abc123!xYz
[WARNING] ⚠️ 安全警告：此默认密码以明文方式暂存于 jwt_secret.json 中，
         请务必登录后立即修改为自定义密码！修改后明文副本将自动删除。
         若不及时修改，存在密码泄露风险。
```

**关键行为规则**：

| 场景 | 日志输出 | 输出级别 |
|------|---------|:--------:|
| 首次安装（初始密码生成时） | 完整密码 + 安全警告 | WARNING |
| 每次重启（用户尚未改密码） | 完整密码 + 安全警告 | WARNING |
| 用户已修改密码后 | **无任何输出** | — |
| 密码重置后 | 新密码 + 安全警告 | WARNING |
| 旧版升级（无可用的明文副本） | 通用警告，提示重置 | WARNING |

**默认密码明文的存储与清除**：
- 明文密码暂存于 `jwt_secret.json` 的 `_temp_plain_password` 字段
- 该字段仅在首次初始化/重置时写入，**用户修改密码后立即清除**
- 清除后内存和磁盘中均不再存在任何形式的明文密码

### Smart 批次回复提示增强（可选）

在 `concurrent_mode=smart` 的基础上，如果同时开启 `enable_smart_batch_reply_hint`：

- Smart 并发不仅会把“当前消息之后紧接着到达的追加消息”一起注入上下文；
- 回复阶段还会动态追加一段 `[系统提示-Smart并发]`；
- 这段提示会明确告诉 AI：当前触发对象仍是主要回复对象，但可以像真人一样自然顺带回应批次中的其他消息；不值得回的消息也可以忽略；
- 这项增强只影响“怎么组织这次回复”，不会改变读空气AI在决策阶段的主体判断，也不会改写保存历史、注意力记录、最近明确回复对象等下游逻辑；
- 该提示只存在于运行时 prompt 中，保存历史前会由 `MessageCleaner` 自动过滤，不会污染普通历史正文。
- ⚠️ Smart 模式当前主要作用于普通消息批次合并与回复阶段提示增强；当主动对话正在占用某个会话时，普通链路在缓存转正/保存前的等待检测，仍然复用通用的 `concurrent_wait_max_loops` 与 `concurrent_wait_interval` 轮询参数，而不是切换成 Smart 专属等待逻辑。
- ⚠️ 选择 `concurrent_mode=smart`，不代表传统并发保护会被关闭；更准确地说，普通对话主线会优先尝试 Smart 批处理，但底层传统并发等待与兜底保护仍然存在，用来处理主动对话占用、等待超时、以及 Smart 未覆盖的异常路径。

之所以不把“主回复对象”直接切换成批次中的其他人，是因为当前保存、注意力、冷却解除、最近明确回复对象等多处链路仍基于“本次主要回复对象 = 当前 event 发送者”的单主体假设。第一版保持这个假设不变，能以最小改动增强多人并发场景下的回复自然度，同时避免牵动无关逻辑。

## 一句话概括

> 本插件**完全接管了 AI 的上下文构建**，它不依赖 AstrBot 平台默认的历史消息和提示词系统，而是自己从头构建发送给 AI 的所有内容（历史消息、系统提示词、当前消息），并在最后一刻替换掉平台准备好的上下文。

## 为什么需要自建上下文？

AstrBot 平台默认的上下文构建是通用的——它把平台收到的历史消息、系统提示词、用户消息直接传给 AI。但对于一个需要"读空气"的群聊插件来说，这远远不够：

| 需求 | 平台默认行为 | 本插件的做法 |
|------|-------------|-------------|
| 消息格式 | 原样传递 | 添加时间戳、发送者名称和ID；多人@场景下会把原始 At 标签增强为 `[At:ID|解析结果]`，在不增加大段说明块的前提下保留提及语义 |
| 历史管理 | 平台统一管理 | 插件自己存储，可独立清除、截断 |
| Bot 自己的回复 | 混在历史中 | 标记为 `【禁止重复-你的历史回复】` |
| 未回复的消息 | 不区分 | 标记为 `【📦近期未回复】`，提供上下文 |
| 单独无信息@消息场景 | 平台无法区分“只是叫出来”还是“想续上文” | 默认启用最近上下文强化，并在通过读空气筛选后按最近明确回复对象给出中性提醒；内部又区分 `contains_ai`（只要空消息里包含@AI就算叫出AI）与 `only_ai`（只能是纯@AI，可重复，不能混他人/全体）两种判定语义 |
| 当前消息 | 与历史混在一起 | 用分隔线明确区分"历史"和"当前" |
| 转发消息 | 仅保留平台原始 Forward 结构/占位 | 在进入上下文前优先解析为单条可读文本，兼容 QQ / OneBot 的嵌套转发 |
| 系统指令 | 平台注入 | 插件注入自己的读空气/回复指令 |
| 窗口缓冲消息 | 不支持 | 等待窗口收集的后续消息单独标注 |
| Smart 批次回复提示 | 无 | Smart 模式下可选地动态提示 AI：主回当前对象，并自然顺带回应批次中的其他消息；该提示只存在于运行时，保存前自动过滤 |

## 核心架构：标记 → 传递 → 拦截 → 替换

整个机制可以用四个字概括：**"偷天换日"**。

> 但当前版本的“换日”并不是靠单一字符串替换硬顶到底，而是采用**优先复用旧成功路径 + 轻量兼容增强 + 失败时保守回退**的方式：能稳定识别时继续按原逻辑清理平台重复注入，识别失败时则优先保证回复链不断、其他插件内容尽量保留，并通过日志显式提示当前处于兼容回退模式。

## 什么被替换了？什么被保留了？

### 被完全替换（平台原始内容被丢弃）

| 内容 | 平台原始值 | 插件替换为 |
|------|-----------|-----------|
| `req.prompt` 主体 | 用户发送的短消息 | 插件格式化的**完整上下文**（历史+指令+当前消息），再合并第三方 prompt 安全补充（带 `[第三方插件片段]` 边界标记） |
| `req.contexts` 主体 | 平台对话历史数组 | 插件自己的 contexts 策略值（普通回复通常为空），再合并差分提取的第三方安全上下文（`req.contexts` 注入型插件的伪造对话、伪工具调用等，带 `[第三方插件注入上下文]` 分隔标记） |
| 平台 LTM 群聊历史 | “You are now in a chatroom...” | **删除**（被插件自己的历史替代） |

> 注意：这里说的是”主体语义被接管”。当前版本不再简单地把第三方对 `req.prompt` / `req.contexts` 的所有改写都一起抹掉；通过差分法识别出的安全长期注入增量，会通过 `_extract_third_party_prompt_additions()` 和 `_extract_third_party_context_additions()` 在恢复阶段追加回最终请求，并附上边界标记防止 AI 混淆不同插件的内容。

### 被合并（平台与插件内容共存）

| 内容 | 处理方式 |
|------|----------|
| `req.func_tool` | 保留平台 `build_main_agent` 注入的框架内置工具（shell/cron/send_message 等），同时合并其他插件注册的 @llm_tool 工具；最终以当前会话 `req.func_tool` 作为工具提醒真源。工具提醒只做提示，不拦截实际工具调用；其中在 `skills_like` 模式下只展示名称/描述，在 `full` 或旧版配置下保持完整参数展示 |

### 被保留（其他插件的贡献不受影响）

| 内容 | 来源 | 保留方式 |
|------|------|----------|
| 向 `system_prompt` 前置的内容 | 满足条件①的插件 | 前置部分作为 prefix，SystemPromptRewriter 识别人格边界后原样保留 |
| 向 `system_prompt` 后置的内容 | 满足条件②的插件 | 后置部分作为 suffix，SystemPromptRewriter 识别人格边界后原样保留 |
| 向 `req.contexts` 注入的伪造对话/协议消息 | 满足条件③的插件 | 快照差分提取，以 `[第三方插件注入上下文]` 标记合并回最终 contexts |
| 向 `req.prompt` 前后追加的长期说明/记忆文本 | 满足条件④的插件 | 短消息基线分割，吸收到固定兼容补充区并以 `[第三方插件片段 N]` 标记 |
| 向 `req.contexts` 注入的伪工具调用 | 满足条件③的插件 | 结构特征识别后追加回最终 contexts |
| `extra_user_content_parts` 中的额外块 | 满足条件⑤的插件 | 原样保留，不做任何修改 |
| 人格设定 | AstrBot 平台 | 作为 `system_prompt` 基础 |
| `extra_user_content_parts` 中的额外块 | 支持该字段的第三方插件 | 保持原样，不被本插件清空 |
| 人格设定 | AstrBot 平台 | 作为 `system_prompt` 基础 |

### 🆕 v1.2.2-hotfix.1 提示词缓存优化

插件现已将大型静态系统指令（`SYSTEM_REPLY_PROMPT`、`SYSTEM_DECISION_PROMPT` 等）追加到 `system_prompt` 中，以提高 AI 服务商的整块缓存命中率。

| AI 调用路径 | 优化方式 | 说明 |
|---|---|---|
| Tier 1（直连 `provider.text_chat()`） | 静态指令搬入 `system_prompt` | DecisionAI.should_reply()：`SYSTEM_DECISION_PROMPT` 从 `prompt` → `system_prompt` |
| Tier 2/3（钩子调用） | 静态指令追加到 `system_prompt` 末尾 | 通过 `PLUGIN_CUSTOM_STATIC_INSTRUCTIONS` extra 传递，`on_llm_request` 钩子在 SystemPromptRewriter 合并后追加 |
| 安全网 | `prompt` 保留原静态前缀 | 钩子失败时 AI 仍收到完整指令，零退化 |

> 对不支持缓存的 AI 服务商：发送内容实质不变，仅位置调整，不影响 AI 理解。

**保留机制的原理**：插件优先沿用旧版的“已知 persona prompt 精确命中 + 删除已知平台 LTM 片段”路径；若平台对 persona 包装或换行做了轻微调整，则退到轻量归一化识别；若仍无法高置信度识别，就进入**保守兼容模式**：优先保留当前 `req.system_prompt`，只在必要时补回插件 persona，并输出 warning 日志提醒可能存在重复提示词。对第三方写入 `req.prompt` / `req.contexts` 的内容，则不会直接原样全吞，而是先做“短消息基线 + 结构特征”安全识别：能确认是长期提示/记忆增量时才并回最终请求；无法确认时宁可跳过，也不把运行时脏内容混入 AI 输入。默认情况下，插件自己的 persona 仍然作为 `system_prompt` 基础放在前面，其他插件附加内容尽量完整保留在后面；这样既最大限度维持旧版效果，又能在未来 AstrBot/第三方插件提示词结构变化时优先保证**主回复链不断、其他插件内容尽量保留、问题可排查**。

### system_prompt 兼容增强与保守回退机制

当前版本对 `system_prompt` 的处理分成三层：

1. **精确命中优先**  
   如果插件自己传入的人格 prompt 能在当前 `req.system_prompt` 中精确找到，就继续按旧逻辑处理；只是额外尝试保留 persona 前后的插件追加内容顺序。

2. **轻量兼容识别**  
   如果精确命中失败，会尝试基于 AstrBot 当前常见的人格包装结构（如 `# Persona Instructions`）做**弱归一化识别**：只兼容空白和换行的轻微变化，不做激进模糊匹配，避免误删其他插件内容。

3. **第三方长期注入吸收**  
   在保持本插件 full prompt / contexts 主导权不变的前提下，再额外扫描第三方在本插件之前写入的 `req.prompt` / `req.contexts`：
   - `req.prompt`：只有能从“短消息基线”里识别出的前缀/后缀型长期说明、记忆文本才会被吸收
   - `req.contexts`：只有结构稳定、能判定为长期记忆型 / fake tool call 型的上下文才会被保留
   - 吸收后的 prompt 文本会统一进入一个固定兼容补充区，并为每一段打上边界标记，避免不同插件提示词混在一起
   - 无法确认安全性的第三方改写会被跳过，并记录 warning

4. **保守回退**  
   如果仍然无法高置信度识别 persona / 平台 LTM 片段，就进入保守兼容模式：
   - 优先保留当前 `req.system_prompt`
   - 尽量不删未知结构的大段文本
   - 只在确定缺失插件 persona 时补回
   - 输出 warning 日志，提示当前可能保留了重复提示词

这意味着：
- **成功路径的语义保持不变**，仍然是“插件 full prompt + 插件 contexts 策略 + 合并后的 system_prompt”。
- **失败路径也不会炸链路**，最差退化为“可能重复，但仍能正常回复”。
- 这是一种**增强版兼容**，不是“万能兼容所有 `on_llm_request` 修改”的承诺。

**如何排查是否进入了回退模式？**
- 开启 `enable_debug_log`：可以看到本次 `system_prompt` 重写采用了哪种策略。
- 未开启 debug 时，如果进入低置信度保守模式，日志仍会输出 warning，说明当前存在平台或第三方插件提示词结构变化，插件已优先选择“不断链”的保守处理。

### 消息保存时的清洗

消息在保存到自定义存储和 `platform_message_history` 表前，会经过 `MessageCleaner` 清洗，**移除所有系统提示词**。正常转正路径（`save_to_official_conversation_with_cache`）和冷群转正路径（`flush_cached_messages_by_params`）均通过 `prepare_cache_for_save` 处理普通缓存消息；冷群转正还会通过 `prepare_window_buffered_for_save` 一并将窗口缓冲消息按时间戳合并转正。所有路径均保留发送者元数据（`sender_id`、`sender_name`、`message_timestamp`），确保写入自定义存储、`platform_message_history` 表和 `conversations` 表时发言人信息正确。

这里的“系统提示词”不仅包括 @ 提示、记忆块、工具提醒，也包括回复AI/主动对话AI在运行时使用的职责说明、元约束、上下文引导等 prompt 正文；这些内容只应参与当次生成，不应作为普通历史正文持久化保存：

```
保存前（AI 看到的）：
  "[系统提示]注意，现在有人在直接@你说话
   [2026-03-13 周四 14:30:00] 张三(ID:111): 你好啊
   === 背景信息 ===
   💭 相关记忆：xxx"

保存后（存入历史的）：
  "张三(ID:111): 你好啊"
```

这确保了下次读取历史时不会出现嵌套的系统提示词。

> **Web 面板会话展示说明**：
> - 「会话管理」与「数据可视化」中的会话列表，统一以复合 `chat_key`（`{platform}_{chat_type}_{chat_id}`，如 `aiocqhttp_group_123456789`）作为 canonical 会话 ID。
> - 插件自定义存储 `chat_history/...` 的聊天记录文件与当前运行时状态（注意力/情绪/概率/缓存等）按 chat_id 匹配后合并到同一个 canonical 条目下。
> - 不同平台的同一 chat_id（如 `aiocqhttp_group_123456789` 和 `gewechat_group_123456789`）会正确拆分为两条独立记录。
> - 官方 `platform_message_history` 仍会同步写入，并在部分场景作为历史读取回退来源，但它不应在 Web 面板里被展示成另一条独立会话。
> - `POST /api/session/clean-ghosts` 可清理无运行时状态的孤立聊天记录文件；前端会话管理页有对应的”清理孤立记录”按钮。

> **工具提醒的保存策略**：工具提醒是运行时 prompt 提示，不属于应当持久化的历史内容。为防止极端情况下混入保存链路，提醒文本会包裹 `[系统提示-工具提醒开始]...[系统提示-工具提醒结束]` 标记，并在 `MessageCleaner` 与 `ContextManager` 的二次清理阶段被兜底过滤。
>
> **skills_like 兼容策略**：当 AstrBot 当前会话使用 `provider_settings.tool_schema_mode=skills_like` 时，工具提醒自动降级为仅展示工具名称与功能描述，不再提前展开参数列表，以尽量避免干扰框架的两阶段工具 schema 暴露流程；若当前为 `full` 模式或旧版 AstrBot 未提供该配置，则维持完整展示。

> **受保护的标记**：以下标记在清洗时会被**保留**，不会被移除：
> - `[PROACTIVE_CHAT]` — 主动对话标记，帮助 AI 理解自己主动发起的对话
> - `[🎯主动发起新话题]` / `[🔄再次尝试对话]` — 主动对话场景标记
> - `[工具调用记录]...[工具调用结束]` — 多工具调用的交错记录
> - `[追加消息上下文]` — 窗口缓冲消息标记，表示该回复生成时已参考了随后收到的追加消息（这些消息在历史中排在回复之后，但实际在回复之前收到）

## 并发处理机制（v1.2.2-hotfix.1+）

### 两种并发处理模式

通过 `concurrent_mode` 配置可以切换两种并发处理策略：

#### legacy 模式（默认）

同一群聊同时收到多条消息时，新消息等待旧消息处理完再依次独立处理，每条消息各自调用 AI 并各自回复。

```
消息A → 处理中（6-8秒 AI 调用）→ 回复A
消息B →     等待中...         → 等待完成 → 处理消息B → 回复B
```

特点：简单可靠，向后兼容，但可能产生重复感（用户发了两条消息，AI 依次回复了两次，内容可能重叠）。

#### smart 模式

AI 决策通过后，在实际调用 AI 生成回复前，检查是否有其他消息同期等待。若有，则将其合并进当前上下文，AI 一次性感知所有消息后回复。

```
消息A → AI决策通过 → 注册待合并池
消息B → AI决策通过 → 注册待合并池
    ↓
消息A 获得并发锁 → Phase 7.5：合并消息B
    → 消息B 标记为"已合并"（不再独立处理）
    → 消息B 内容追加到 formatted_context 的 window_buffered 区域
    ↓
AI 一次性感知 A+B → 生成统一回复
消息B 的处理协程：检测到 is_merged → 直接返回（不重复调用 AI）
```

特点：避免重复感，更符合人类对话习惯；仅能合并"几乎同时到达"的消息（时间差在 AI 决策耗时内约 5-10 秒）。追加消息区域现在会复用群聊等待窗口（GWW）的展示增强逻辑：基础 `@` 解析会展开为 `[At:ID|解析结果]`，`@全体` 会补充说明，持久化戳一戳事件文本也会一并显示给 AI，但运行时的主消息专用 `[系统提示]` / `【@指向说明】` 仍不会被硬塞进追加消息区。

### 主动对话与普通对话的协调（v1.2.2-hotfix.1+）

#### 普通对话冷静期（`proactive_normal_reply_cooldown`）

AI 通过普通对话流程回复了群里某人后，在 `proactive_normal_reply_cooldown`（默认 60 秒）内，主动对话系统不会触发新的主动发言。

**解决的问题**：旧版本中，主动对话计时器只看 `last_bot_reply_time`（无论是主动还是普通回复），因此 AI 刚刚正常回复完用户消息，仍可能立刻触发主动发言，造成「刚聊完又来找话说」的奇怪体验。

#### 吐槽系统感知普通活跃度（`generate_complaint_prompt`）

吐槽系统在生成吐槽文案时，会感知近 10 分钟内是否有过普通对话回复：

| 情况 | 策略 |
|------|------|
| 普通对话活跃（近 10 分钟内有普通回复） | 轻/中度吐槽概率 ×0.5，强烈吐槽概率 ×0.7；文案改为「感觉大家都在聊但没来找我」 |
| 普通对话不活跃（群里真的没人聊） | 原有文案：「连续主动说了 N 次都没人回」 |

**解决的问题**：旧版本中，AI 刚刚通过普通流程回复了很多人，然后主动对话触发了吐槽（连续 N 次无人回应），结果 AI 抱怨「群里没人理我」，与实际情况矛盾。

## 并发处理机制

### 两种并发处理模式

通过 `concurrent_mode` 配置可以切换两种并发处理策略：

#### legacy 模式（默认）

同一群聊同时收到多条消息时，新消息等待旧消息处理完再依次独立处理，每条消息各自调用 AI 并各自回复。

```
消息A → 处理中（6-8秒 AI 调用）→ 回复A
消息B →     等待中...         → 等待完成 → 处理消息B → 回复B
```

特点：简单可靠，向后兼容，是最兜底的并发保护模式；但可能产生逐条回复的重复感。

#### smart 模式

smart 模式不再按“谁先跑到某个异步点”决定主消息，而是先为消息登记到达顺序，再让最早到达的消息担任 anchor。anchor 在进入读空气 AI 前吸收后续已准备好的消息，DecisionAI 与 ReplyAI 看到的是同一批上下文。

```
消息A(先到) -> arrival_seq=1
消息B(后到) -> arrival_seq=2
    ↓
A 成为 anchor
    -> 决策前吸收 B
    -> B 标记为 consumed
    -> DecisionAI / ReplyAI 同时看到 A + 追加消息B
    ↓
统一生成一次回复
```

追加消息区域沿用窗口缓冲消息的上下文表达方式：这些消息位于当前消息之后，可能来自不同用户；若 `include_sender_info` / `include_timestamp` 开启，则会保留对应的发送者与时间信息。

### Smart 与群聊等待窗口（GWW）的关系

- GWW 负责”同一用户短时间拆分消息”的补收集
- Smart 负责”同群多条并发消息按真实顺序批处理”
- 两者**可以协作，但互不依赖**
- 即使不开 GWW，Smart 也能独立工作
- 进入 GWW 的消息不会再进入 Smart 批处理流程

### 等待窗口令牌绑定与回落机制（v1.2.2-hotfix.1+）

每个等待窗口创建时分配唯一递增令牌（`gww_token`）。窗口期内被拦截的每条消息将携带该令牌写入缓存。当锚点消息经读空气AI判定”不回复”后，`convert_window_buffered_to_regular` 按三层过滤精确转换该窗口批次的 `window_buffered=True` 消息为普通缓存：

| 过滤层 | 键 | 作用 |
|--------|-----|------|
| 会话隔离 | `chat_id` | 不同群聊/私信互不影响 |
| 用户隔离 | `sender_id` | 仅转换当前窗口所属用户的消息 |
| 窗口批次隔离 | `gww_token` | 同一用户多个并发窗口之间令牌不同，互不污染 |

回落后的消息与普通缓存行为完全一致：参与 Phase-1 转正、冷群转正、按原始时间戳统排顺序。这彻底消除了旧窗口消息残留 `window_buffered` 标记导致的下次回复上下文顺序错乱问题。Smart 批次的回落（`smart_batch_snapshots` → `pop(“window_buffered”)` → 注入缓存）与 GWW 窗口回落（`gww_token` 精确匹配）分别处理各自的消息来源，互不干扰。

### 普通对话 / 主动对话 / 冷群转正 的互斥

群聊内部现在有一层会话级 flow owner 协调：

- `normal`：普通对话回复链
- `proactive`：主动对话链
- `idle_flush`：冷群缓存自动转正链

规则：
- 普通对话优先级最高
- 主动对话开始前会等待普通对话释放当前会话
- 冷群转正只在会话空闲时执行
- 这是内部强制生效的实现，不新增配置项；如果未开启主动对话，则相关互斥逻辑不会实际介入

### 动态提示词与历史清理

Smart 追加消息、多用户提醒、主动对话预判断中的并发补充说明，均采用**动态注入**：

- 不直接写死到总系统提示词常量中
- 只在对应场景发生时才拼接
- 保存用户消息 / AI 回复 / 缓存转正时会统一通过 `MessageCleaner` 清理
- 不影响图片转文字 AI 及其相关配置语义

## 决策 AI vs 回复 AI：两条独立的 AI 调用路径

本插件有**两次独立的 AI 调用**，它们走完全不同的路径：

```
┌──────────────────────────────────┐   ┌──────────────────────────────────┐
│     决策 AI（"读空气"）          │   │     回复 AI（"生成回复"）        │
├──────────────────────────────────┤   ├──────────────────────────────────┤
│                                  │   │                                  │
│ 调用方式：                       │   │ 调用方式：                       │
│   provider.text_chat()           │   │   event.request_llm()            │
│   ↑ 直接调用 AI 提供商           │   │   ↑ 走平台标准流程               │
│   ↑ 不经过任何 Hook              │   │   ↑ 经过 Hook 链                 │
│   ↑ 不受其他插件影响             │   │   ↑ 受本插件 on_llm_request 拦截 │
│                                  │   │                                  │
│ 系统提示词：                     │   │ 系统提示词：                     │
│   内置的决策判断指令             │   │   人格设定 + 其他插件内容        │
│   + decision_ai_extra_prompt     │   │   + 回复指令                     │
│   + 注意力/情绪/疲劳状态        │   │   + reply_ai_extra_prompt        │
│                                  │   │                                  │
│ 上下文：                         │   │ 上下文：                         │
│   近期消息摘要（较短）           │   │   完整格式化历史（完整）         │
│                                  │   │   + 记忆 + 工具 + 窗口消息      │
│                                  │   │                                  │
│ 输出：                           │   │ 输出：                           │
│   YES / NO                       │   │   AI 的回复文本                  │
│                                  │   │                                  │
│ AI 提供商：                      │   │ AI 提供商：                      │
│   decision_ai_provider_id        │   │   AstrBot 默认提供商             │
│   （可单独配置轻量模型）         │   │                                  │
│                                  │   │                                  │
└──────────────────────────────────┘   └──────────────────────────────────┘
```

### 私信链路的对应关系

私信模块虽然仍处于开发测试阶段，但 AI 调用类别与群聊主链路保持同类对齐：

- **私信回复生成**：与群聊回复生成一样走 `event.request_llm()`，先传短消息/占位 prompt，再由 `on_llm_request` 恢复完整上下文
- **私信主动对话生成**：与群聊主动对话一样走 `ProviderRequest + OnLLMRequestEvent + provider.text_chat()` 这条兼容链路
- **私信图片转文字**：与群聊图片转文字一样保持 `provider.text_chat()` 直连，不进入回复 Hook 链

也就是说，这里并不是把所有 AI 调用统一成一种方式，而是让**每一类 AI 继续走它本来对应的那条链路**。

### 决策 AI / 回复 AI 的最近上下文增强

除了原本的完整历史上下文外，当前实现还额外引入了两层“最近上下文”增强，但它们彼此解耦：

1. **普通概率路径的注意力冷却补丁**
   - 只作用于非@AI、非关键词触发的普通概率路径消息
   - 当一条消息看起来不是发给AI时，不会立刻把高注意力用户推进正式冷却，而是先进入“未接续谈保护”（pending reconnect）
   - 未接续谈保护只观察**同一用户自己的后续消息**，不受群里其他人插话干扰
   - 如果该用户紧接着又发普通消息，系统会给他一个最低概率保护，减少“上一条没接住，下一条普通消息也被一起误伤”的情况
   - 如果该用户已经不在当前会话的注意力追踪列表中，则其 pending / active 冷却状态会被同步移除，不再继续推进到正式冷却
   - “读空气未回复衰减”已从冷却本体中解耦：未开启冷却时可独立生效；开启冷却时，只有正式冷却后才执行一次注意力衰减
   - 冷却状态改为运行态内存数据，插件或平台重启后会自然清空，不再长期持久化

2. **单独的、不包含任何信息的 @ 消息默认上下文强化**
   - 只在“仅@AI且没有文字/图片/关键词等其他内容”时触发
   - 不依赖注意力冷却状态，而是独立参考“最近一次明确回复对象”
   - 系统会先走完前置过滤与读空气筛选，只有最终决定回复时，才在回复阶段追加这层提醒
   - 若当前发送这条消息的人与最近一次明确回复对象相同，且仍在关联窗口内，则给AI一段**中性**提示：优先参考最近上下文，但不要强行续话
   - 若不是同一人，或时间 / 消息数任一窗口超限，则降级为更中性的自由判断提示
   - 关联窗口阈值默认启用，可通过 `single_at_message_reply_link_max_messages` 与 `single_at_message_reply_link_max_seconds` 配置；两者会同时生效

这意味着：
- **读空气AI** 不会被这层提醒提前干扰，仍只负责判断该不该回
- **回复AI** 会在真正需要回复时更自然地参考最近上下文，但不会死盯上一段历史
- 这层机制只是“回复阶段的增强判断线索”，不会把 AI 锁死在必须续接上文的行为里
- 当 `concurrent_mode=smart` 且开启 `enable_smart_batch_reply_hint` 时，回复阶段还会动态追加一段 `[系统提示-Smart并发]`，提醒 AI 主回当前对象，并可自然顺带回应批次中的其他消息
- 这类 Smart 提示同样属于运行时 prompt，会在保存前自动过滤，不会进入普通历史正文

## 🆕 AI 错误处理机制：智能识别与零副作用跳过

### 核心设计理念

当 AI 调用失败时（如服务商 502、网络超时等），系统应**明确区分「外部故障」和「代码问题」**，并确保**错误不会触发任何副作用**（打分/概率调整/冷淡期/沉默计时器重置）。

### 错误格式化器 (`ai_error_formatter.py`)

所有 AI 调用的异常都通过 `format_ai_error()` 统一处理，自动识别 5 类错误：

| 错误类型 | 识别特征 | 日志输出 |
|---------|---------|---------|
| **HTML 网关错误** (502/503/504) | 响应体包含 `<html>` / `cloudflare` / `Bad Gateway` 等 | `⚠️ AI 服务商故障 HTTP 502：... → 这不是插件代码的问题` |
| **上游模型空输出** | 包含 `upstream_empty_output` / `empty output` / `no usable output` 等关键词 | `⚠️ 上游模型返回空输出：模型/中转接口这次没有返回可用内容` |
| **HTTP 状态码错误** (400-599) | 异常信息包含标准状态码 | `⚠️ AI 服务商故障（HTTP 500）` 或 `⚠️ 请求参数/配置问题（HTTP 400）` |
| **网络连接问题** | 包含 `timeout` / `connection refused` / `DNS` 等关键词 | `⚠️ 网络问题（请求超时）` |
| **其他异常** | 不匹配以上模式 | 普通错误信息（超长时自动截断至 300 字符） |

### 零副作用跳过原则

以下场景中，AI 调用失败被视为**本次操作未发生**，不触发任何状态变更：

| 场景 | 影响的机制 | 处理方式 |
|------|----------|---------|
| **主动对话生成 AI 调用失败** | 打分、概率提升、冷淡期、沉默计时器 | 显式 `return`，不执行步骤 8（`record_bot_reply` + `activate_temp_probability_boost`） |
| **主动对话发送/保存阶段出错** | 同上 | 外层 `except` 仅记录日志，不触发副作用 |
| **主动对话 AI 预判断调用失败** | 沉默计时器重置 | 不更新 `last_bot_reply_time`，快速进入下一次检测 |
| **主动对话 AI 预判断超时** | 同上 | 同上 |
| **读空气（决策 AI）调用失败** | — | 返回空字符串，视为「未判断」，不影响后续流程 |
| **普通群聊回复：LLM 返回空文本** | 成功回复后的冷却解除、最近明确回复对象、AI 回复历史保存 | 不保存 AI 回复；但仍继续保存当前用户消息、待转正缓存以及窗口缓冲 Phase-2，避免上下文断裂 |
| **图片转文字 AI 调用失败** | — | 跳过该张图片，继续处理其余图片 |

> **对比：正常业务逻辑 vs 异常跳过**
>
> - **AI 预判断返回 `no`（判断不通过）** → 这是正常的业务结果，会更新 `last_bot_reply_time` 重置沉默计时器 ✅
> - **AI 预判断抛出 502 异常** → 这是外部故障，**不更新**任何状态 ❌→❌
> - **普通回复真正生成出文本** → 正常保存 AI 回复，并继续执行冷却解除、最近明确回复对象等成功回复副作用 ✅
> - **普通回复最终为空文本** → 视为“这次没有真正回出去”；只补做用户消息与缓存转正保存，不补做成功回复副作用 ❌→✅（仅上下文保存）

## 与其他插件的兼容性

### 消息处理链兼容性（v1.2.1+）

本插件的消息处理器（`on_group_message`、`on_private_message`）使用 **`priority=-1`**，低于 AstrBot 默认优先级（0），确保其他插件的消息处理器**先于本插件执行**。

如果其他插件（如 `better_reminder`、`llm_poke` 等）已经对当前消息发送了回复，本插件会检测到 `event._has_send_oper` 标记并**自动跳过 AI 处理**，避免重复回复。跳过时仍会将消息缓存到上下文池，确保 AI 不丢失上下文。

```
消息处理链执行顺序（AdapterMessageEvent）
═══════════════════════════════════════════════════

优先级 sys.maxsize-1   command_filter_handler（本插件）
                       ├─ 检测指令消息（/help、!reset 等）
                       └─ 标记后返回，不发送回复，不阻止传播
                           ↓
优先级 0（默认）       其他插件的消息处理器
                       ├─ better_reminder.on_message
                       ├─ self_learning.on_message（只收集数据，不发送）
                       ├─ 其他插件...
                       └─ 如果某个插件发送了回复 → _has_send_oper = True
                           ↓
优先级 -1              on_group_message / on_private_message（本插件）
                       ├─ 检查 _has_send_oper
                       │   ├─ True → 缓存消息到上下文池，跳过 AI 处理
                       │   └─ False → 正常处理（概率→读空气→回复）
                       └─ 处理完毕
```

> **注意**：AstrBot 框架在每个 handler 之间调用 `event.clear_result()`，会清除 `stop_event()` 状态。因此 `stop_event()` 无法跨 handler 传播。`_has_send_oper` 标记不受 `clear_result()` 影响，是唯一可靠的跨 handler 通信方式。

> **与 LLM 钩子链无关**：消息处理链（`AdapterMessageEvent`）和 LLM 钩子链（`OnReqLLMEvent`）使用独立的优先级排序。本插件消息处理器的 `priority=-1` 不影响 `on_llm_request` 钩子的 `priority=-1`，两者在不同的事件链中各自独立排序。

### 如何判断一个插件是否兼容

判断方法很简单：**查看该插件的 `on_llm_request` 钩子，看它往 `req` 的哪个字段、以什么方式注入了内容。** 只要匹配以下任一条件，就可以兼容。

**自查三步：**

1. **找到该插件的源码**，搜索 `on_llm_request`，看它是否注册了这个钩子。如果没有 → 兼容（条件⑦）。
2. **看它修改了 `req` 的哪些字段**（`system_prompt` / `contexts` / `prompt` / `extra_user_content_parts`），以及是**前置、后置还是新增**。
3. **看装饰器的 `priority=` 参数**：绝大多数插件是 0 或正数，在本插件之前执行 → 正常兼容；如果 < -1（极少见），则在本插件之后执行，内容可能丢失。

#### 兼容条件（满足任一即可）

| 条件 | 判断依据 | 保留机制 |
|------|---------|---------|
| ① 向 `req.system_prompt` **前面插入**了内容 | 插件代码中有类似 `req.system_prompt = xxx + req.system_prompt` 或 `req.system_prompt = f”{xxx}\n{req.system_prompt}”` 的写法 | SystemPromptRewriter 识别出人格边界后，前面的内容作为 prefix 原样保留 |
| ② 向 `req.system_prompt` **后面追加**了内容 | 插件代码中有类似 `req.system_prompt += xxx` 或 `req.system_prompt = req.system_prompt + xxx` 的写法 | SystemPromptRewriter 识别出人格边界后，后面的内容作为 suffix 原样保留 |
| ③ 向 `req.contexts` 中**新增了消息** | 插件代码中有类似 `req.contexts.append(...)` 的写法，且追加的是有 `role`/`content` 结构的 dict | 快照差分提取 → 以 `[第三方插件注入上下文]` 标记合并回最终 contexts |
| ④ 向 `req.prompt` **前面或后面追加**了内容 | 插件代码中修改了 `req.prompt`，在原用户消息前后加了额外的说明/记忆文本 | 短消息基线分割 → 吸收到 `[第三方插件补充信息]` 固定补充区 |
| ⑤ 向 `req.extra_user_content_parts` **追加**了内容 | 插件代码中有类似 `req.extra_user_content_parts.append(...)` 的写法 | 原样保留，不做任何修改 |
| ⑥ 不修改 `req`（自己调用 AI） | 插件绕过 `on_llm_request`，直接用 `provider.text_chat()` 发起请求 | 互不干扰 |
| ⑦ 不注册 `on_llm_request` 钩子 | 插件不参与 LLM 请求的上下文构建 | 没有冲突 |

#### 可能冲突的情况

| 情况 | 原因 | 后果 |
|------|------|------|
| 直接整体重写 `req.prompt` | 无法从短消息基线中提取安全增量 | 该插件内容可能被跳过 |
| 塞入结构异常/字段缺失的 contexts | 差分法只吸收有 `role`/`content` 的合法 dict | 异常内容会被丢弃 |
| 有自己的 LTM/历史管理且同时注入平台历史 | 两套历史系统语义重叠 | AI 可能看到重复或矛盾的历史 |
| 本插件接管后运行时（priority < -1）才执行 | 在本插件差分快照之后才修改 req | 注入内容不会被捕获 |
| 依赖平台默认上下文格式 | 本插件主导最终 prompt/contexts 语义 | 该插件可能无法按原设计工作 |

#### 优先级风险说明

`on_llm_request` 钩子的**优先级决定了执行顺序**（数值越大越先执行）：

| 优先级 | 执行时机 | 对本插件的影响 |
|--------|---------|---------------|
| **> 0**（如 100000） | 在本插件**之前**执行 | ✅ 修改被快照捕捉，差分法正常保留 |
| **0（默认）** | 在本插件**之前**执行 | ✅ 同上，差分法正常保留 |
| **-1** | 本插件自身 | — |
| **< -1** | 在本插件**之后**执行 | ⚠️ 快照已拍完，此后的修改**不会被差分法捕获**，注入内容可能丢失 |

> **排查方法**：如果你不确定某个插件的优先级，在它的 `on_llm_request` 装饰器上查看 `priority=` 参数。未显式指定时通常为 0。绝大多数第三方插件不会使用 < -1 的优先级，所以这个风险实际很少触发。如果真的遇到了，可以联系该插件作者调整优先级，或在 debug 日志中确认是否丢失。

#### system_prompt 识别的三种路径与风险

SystemPromptRewriter 在合并 system_prompt 时有三种策略：

| 策略 | 触发条件 | 保留效果 | 风险 |
|------|---------|---------|------|
| **精确命中** (confidence=high) | 插件自己的人格 prompt 在 `req.system_prompt` 中被精确找到 | prefix + persona + suffix 完整保序 | 几乎没有风险 |
| **轻量归一化** (confidence=medium) | 平台对 persona 做了轻微包装（如 `# Persona Instructions`），仅空白/换行有差异 | 同样保序，但会输出提示告知走了归一化路径 | 极小概率误识别边界 |
| **保守回退** (confidence=low) | 无法高置信度识别 persona 边界 | 优先保留当前 system_prompt，只去掉已知 LTM 片段，补回插件 persona | **宁重复不缺漏**，可能保留部分平台注入的重复内容，但主回复链不断 |

> 如果日志中出现 `[兼容回退]` warning，说明当前 system_prompt 结构与预期有差异，插件已优先选择"不断链"。此时 AI 可能看到少量重复的平台提示词，但第三方插件的注入内容仍然会被尽量保留。

#### 平台 LTM 格式变更的风险

当前平台 LTM 的识别依赖正则模式表（匹配 `"You are now in a chatroom..."` 等已知格式）。如果 AstrBot 未来版本改写了这些注入文案：

- **能匹配上** → 正常删除，避免与插件自建历史重复
- **匹配不上** → 正则失效，平台 LTM 片段被保留（可能造成 system_prompt 中同时存在平台历史和插件历史）

> 这里的设计原则是**宁重复不缺漏**：删不掉平台历史最多让 AI 看到两份上下文（通常不影响回复质量），但误删第三方插件的注入会导致功能异常。

#### 流程图

```
该插件 on_llm_request 钩子修改了 req 的什么？
    │
    ├─ req.system_prompt 前置/后置？
    │   └─ ✅ 兼容（条件①②）
    │
    ├─ req.extra_user_content_parts 追加？
    │   └─ ✅ 兼容（条件⑤）
    │
    ├─ req.prompt 前后追加？
    │   ├─ 仅前后追加说明/记忆 → ✅ 兼容（条件④）
    │   └─ 整体改写成另一套对话 → ⚠️ 可能丢失
    │
    ├─ req.contexts 追加？
    │   ├─ 有 role/content 的合法 dict → ✅ 兼容（条件③）
    │   └─ 结构异常/运行时中间态 → ❌ 会被丢弃
    │
    ├─ 不修改 req（自己调 AI）？
    │   └─ ✅ 兼容（条件⑥）
    │
    └─ 不注册 on_llm_request？
        └─ ✅ 兼容（条件⑦）
```

#### 最简单的验证方法

开启 `enable_debug_log`，在日志中查看：
- `system_prompt` 走了哪条重写策略
- 是否吸收了第三方 prompt 补充（`[兼容增强] 已吸收 N 段第三方提示词`）
- 是否保留了第三方 contexts 补充（`[兼容增强] 已保留 N 条第三方上下文注入`）

## Hook 执行顺序详解

AstrBot 的 Hook 系统按**优先级从高到低**执行。本插件使用 `priority=-1`，确保在**最后执行**，这样它可以看到所有其他插件追加的内容，并优先按“旧成功路径 → 轻量兼容识别 → 保守回退”的顺序处理：

```
on_llm_request Hook 执行链
═════════════════════════════════════════════════

优先级 100000   向 system_prompt 后置追加的插件
                ├─ 向 system_prompt 后面追加状态规则 + 输出协议
                ├─ 向 contexts 注入伪造对话示例 + 协议强制消息
                └─ 不修改 prompt
                    ↓
优先级 0（默认）向 system_prompt 前置 / 向 prompt 注入 / 向 extra 注入 等插件
                ├─ 前置注入型: 向 system_prompt 前面插入规则块
                ├─ prompt/contexts 注入型: 向量检索后按模式注入
                │   · system_prompt 模式 → 追加记忆到 system_prompt
                │   · user_message 模式 → 在 prompt 前后追加
                │   · fake_tool_call 模式 → 向 contexts 注入伪工具调用
                ├─ extra 注入型: 向 extra_user_content_parts 注入
                │   <context> 标签
                │   回退时追加到 system_prompt
                └─ 不修改 prompt（extra 注入型主要通过 extra）
                    ↓
优先级 0        AstrBot 平台 LTM（长期记忆）
                ├─ 在 system_prompt 末尾追加：
                │   "You are now in a chatroom.
                │    The chat history is as follows:
                │    [平台存储的群聊历史]"
                ├─ 设置 req.contexts = [平台对话上下文]
                └─ 不修改 prompt
                    ↓
优先级 -1       ★ 本插件 on_llm_request ★
                │
                │  此时 req.system_prompt 的内容可能接近：
                │  "[人格] + [后置追加的内容] + [LTM 群聊历史]"
                │  也可能因平台版本或第三方插件包装差异出现轻微变化
                │
                ├─ 1. 优先尝试精确命中插件自己的人格 prompt
                │
                ├─ 2. 若精确命中失败，再尝试轻量兼容识别
                │     （如 `# Persona Instructions` 包装 + 空白归一化）
                │
                ├─ 3. 对已知平台 LTM 片段做模式表清理
                │     （成功时移除，避免与插件自建历史重复）
                │
                ├─ 4. 若仍无法高置信度识别
                │     → 进入保守兼容模式
                │     → 优先保留当前 system_prompt
                │     → 输出 warning，不中断主回复链
                │
                ├─ 5. 基于短消息基线尝试吸收第三方 prompt 长期补充
                │     （只保留安全增量，不直接原样继承整段 prompt）
                │
                ├─ 6. 基于结构特征尝试保留第三方长期 contexts
                │     （如 fake tool call / 记忆型上下文）
                │
                ├─ 7. 替换 req.prompt = 插件的完整上下文
                │     + 第三方安全补充区
                │
                ├─ 8. 恢复 req.contexts = 插件既有策略值
                │     + 第三方安全长期上下文
                │
                └─ 9. 清除 event.extra 中的标记

═════════════════════════════════════════════════
最终结果：
  system_prompt = [前置注入的内容] + 插件人格 + [后置注入的内容] + [prompt/contexts 注入的内容]
                  + 静态指令 + Skills + TOOL_CALL_PROMPT（正常路径下已去掉已知 LTM）
  prompt        = 插件格式化的完整上下文 + 安全识别出的第三方 prompt 补充
                  （带 [第三方插件片段 N] 边界标记）
  contexts      = 插件既有策略值 + 安全识别出的第三方长期上下文
                  （带 [第三方插件注入上下文] 分隔标记）
  extra_user_content_parts = 原样保留（extra 注入型插件内容不受影响）
```

> **重要说明**：保守回退模式的目标是“不断链、可排查、尽量保留其他插件内容”，不是“保证任何未来版本 AstrBot 都能 100% 无重复地清理提示词”。如果平台未来彻底改写了相关注入文案，本插件会优先保证回复继续可用，并通过日志明确提示当前兼容识别失败。

> **边界说明**：这里的兼容增强是“安全吸收长期注入内容”，不是“让第三方对 `req.prompt` / `req.contexts` 的任意改写都原样生效”。若第三方直接整体改写 prompt，或塞入与长期记忆无关的运行时中间态，本插件会宁可跳过，也不会贸然并入最终请求。

## 为什么用"短消息"调用再替换？

你可能会问：既然最终要替换成完整上下文，为什么不一开始就直接传完整内容？

这是因为 `event.request_llm()` 调用后会触发 Hook 链，其中一些插件（如 LivingMemory 记忆插件）需要**短消息**来做向量检索：

```
event.request_llm(prompt = "你们觉得去哪好？")   ← 短消息
        ↓
LivingMemory Hook（优先级 0）：
  用 "你们觉得去哪好？" 做向量搜索
  找到相关记忆，追加到 system_prompt
        ↓
本插件 Hook（优先级 -1）：
  LivingMemory 追加的记忆 → 被保留在 system_prompt 中 ✅
  替换 prompt = 完整上下文  ← 此时再换成完整的
```

如果一开始就传完整上下文（几百条历史消息），LivingMemory 会用整个历史去做向量搜索，效果很差而且浪费算力。

而当前版本比旧版多了一层兼容增强：
- 如果 LivingMemory 或其他插件在这一步之后，又把长期记忆说明前缀/后缀追加到了 `req.prompt`，插件会尝试把这些说明吸收到一个**固定、有边界标记**的兼容补充区，而不是直接把整段 prompt 生硬覆盖回去。
- 如果它们把长期记忆结果以 fake tool call / 稳定 `contexts` 结构写进 `req.contexts`，插件也会尝试把这些内容安全追加回最终上下文。
- 这样做的目标不是“让所有第三方随便改什么都生效”，而是让**长期提示词尽量被 AI 看见，同时版式清楚、边界明显、不会把脏的运行时中间态混进来**。

## 为什么必须开启平台的"群聊上下文感知"？

> 这是最容易让人困惑的问题：既然本插件已经完全接管了上下文构建，为什么还要在 AstrBot 平台设置里开启**聊天记忆增强（群聊上下文感知）**？

### 原因：插件需要读取平台存储的历史消息

本插件的上下文来源分两部分：

```
┌──────────────────────────────────────────────────────────────┐
│              插件的历史消息读取策略                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  优先级 1：读取平台官方存储                                  │
│  ┌────────────────────────────────────────────────┐         │
│  │ platform_message_history 表（SQLite 数据库）    │         │
│  │                                                 │         │
│  │ • 由 AstrBot 平台的 LTM 系统写入              │         │
│  │ • 数据持久化，重启后不丢失                     │         │
│  │ • 包含完整的群聊消息记录                       │         │
│  │ • 插件通过 context.message_history_manager     │         │
│  │   读取这个表中的数据                           │         │
│  └────────────────────────────────────────────────┘         │
│         ↑                                                    │
│         │  如果读取成功 → 使用这个                           │
│         │  如果读取失败 ↓                                    │
│                                                              │
│  优先级 2：回退到插件自定义存储                              │
│  ┌────────────────────────────────────────────────┐         │
│  │ custom_storage（插件自己的 JSON 文件）          │         │
│  │                                                 │         │
│  │ • 由插件自己写入和管理                         │         │
│  │ • 作为备用数据源                               │         │
│  │ • 受 custom_storage_max_messages 限制          │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**关键点**：`platform_message_history` 表的数据是由平台的 LTM 系统（聊天记忆增强）在 `on_message` 事件中写入的。**如果你不开启"群聊上下文感知"，平台就不会把群聊消息写入这个表，插件就读不到历史消息。**

### 平台 LTM 的双存储机制

AstrBot 平台的 LTM 系统实际上有**两套独立的存储**：

```
┌─────────────────────────────────────────────────────────────────┐
│         AstrBot 平台 LTM 系统的两套存储                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  存储 A：内存中的 session_chats（临时）                         │
│  ┌──────────────────────────────────────────────────┐           │
│  │ dict: { "平台:群聊:群号": ["消息1", "消息2", ...] }│          │
│  │                                                    │          │
│  │ • 纯内存，重启后清空                              │          │
│  │ • 受 group_message_max_cnt 限制（默认 300）       │          │
│  │ • 格式：[发送者/HH:MM:SS]: 消息内容              │          │
│  │ • 用途：on_llm_request 中注入到 system_prompt     │          │
│  │         "You are now in a chatroom..."            │          │
│  │                                                    │          │
│  │ ★ 本插件在 on_llm_request 中会删除这部分 ★      │          │
│  │   （因为插件用自己的格式化历史替代了它）          │          │
│  └──────────────────────────────────────────────────┘           │
│                                                                  │
│  存储 B：数据库中的 platform_message_history（持久）            │
│  ┌──────────────────────────────────────────────────┐           │
│  │ SQLite 表：data/data_v4.db                        │          │
│  │                                                    │          │
│  │ • 持久化存储，重启后保留                          │          │
│  │ • 无最大条数限制（平台不会自动清理）              │          │
│  │ • 包含完整消息结构（发送者、时间、内容等）        │          │
│  │ • 用途：供其他插件读取群聊历史                    │          │
│  │                                                    │          │
│  │ ★ 本插件从这里读取历史消息 ★                    │          │
│  │   然后按自己的格式重新组装                        │          │
│  └──────────────────────────────────────────────────┘           │
│                                                                  │
│  两套存储互相独立，不同步                                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

所以实际上发生的事情是：

1. 平台 LTM 收到消息 → 写入 session_chats（内存）+ platform_message_history（数据库）
2. 本插件需要回复时 → 从 platform_message_history（数据库）读取历史
3. 平台 LTM 在 on_llm_request 中注入 session_chats 内容到 system_prompt
4. 本插件在 on_llm_request 中**删除**平台注入的内容，**替换**成自己格式化的历史

**一句话总结：插件需要平台的 LTM 帮它"收集"消息并存到数据库，但不需要平台帮它"注入"上下文。收集是收集，注入是注入——收集靠平台，注入靠自己。**

### group_message_max_cnt 与插件的关系

| 配置项 | 属于 | 影响 |
|--------|------|------|
| 平台 `group_message_max_cnt`（默认 300） | AstrBot 聊天记忆增强 | 只限制**内存中 session_chats** 的条数。本插件会删掉这部分内容，所以**对本插件无直接影响** |
| 插件 `max_context_messages`（默认 -1） | 本插件 | 限制插件从数据库读取后**传给 AI 的最大条数** |
| 插件 `custom_storage_max_messages`（默认 500） | 本插件 | 限制插件**自定义备用存储**的最大条数 |

**所以 `group_message_max_cnt` 设置多少都无所谓？**

不完全是。虽然这个值不影响插件读取的历史条数（插件读的是数据库，不受这个限制），但它影响的是平台 LTM 的**内存占用**。设置太大会浪费内存（因为本插件用不上这些内存中的数据），设置太小也没问题。

**推荐设置：**

```
group_message_max_cnt: 10 ~ 50
```

理由：
- 这个值只影响平台自己的内存缓存（session_chats），本插件会删掉它注入的内容
- 设成较小值（如 10-50）可以**节省内存**，因为本插件不依赖它
- 不要设成 0 或关闭，否则平台 LTM 可能不写入 platform_message_history 表
- 保留少量条数是为了确保 LTM 系统正常运行并持续向数据库写入消息

## 为什么必须开启"自动理解图片"？

### 平台的图片处理流程

当群聊中有人发送图片时，AstrBot 平台的 LTM 系统会把消息存入 `platform_message_history`。问题是：**图片怎么存？**

```
                  群友发了一张图片
                        ↓
              平台 LTM handle_message()
                        ↓
              ┌─────────┴─────────┐
              ↓                   ↓
    自动理解图片: 关闭      自动理解图片: 开启
              ↓                   ↓
    存入数据库的内容：       存入数据库的内容：
    "[Alice/14:23:45]:      "[Alice/14:23:45]:
     [Image]"                [Image: 一只橘猫
                              趴在桌子上睡觉]"
              ↓                   ↓
    插件读取到的是：         插件读取到的是：
    "[Image]"               "[Image: 一只橘猫
     ↓                       趴在桌子上睡觉]"
    AI 只知道"有图"          ↓
    但不知道图里是什么       AI 知道图片内容！
```

**关键点**：平台的"自动理解图片"功能会在 LTM 存储消息时，调用 AI 对图片进行描述，并把描述文本存入 `platform_message_history`。当本插件后续从数据库读取历史时，就能拿到带图片描述的消息。

### 那本插件自己的图片处理呢？

本插件有自己的 `enable_image_processing` / `image_to_text_provider_id` 图片处理功能，但两者作用在**不同的时间点**：

```
┌──────────────────────────────────────────────────────────────┐
│            两套图片处理的作用时机对比                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  时机 1：消息被存入历史时（平台"自动理解图片"）             │
│  ┌────────────────────────────────────────────────┐         │
│  │ 群友发图 → 平台 LTM → 调用 AI 描述图片       │         │
│  │         → 存入 "[Image: 描述文本]" 到数据库   │         │
│  │                                                │         │
│  │ 影响：历史消息中的图片有没有描述               │         │
│  │ 没开启 → 历史中只有 "[Image]"，AI 看不懂     │         │
│  │ 开启   → 历史中有 "[Image: 描述]"，AI 能理解 │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  时机 2：当前消息被处理时（插件"图片处理"）                 │
│  ┌────────────────────────────────────────────────┐         │
│  │ 群友发图 → 插件概率筛选通过                   │         │
│  │         → 插件调用 AI 描述图片                │         │
│  │         → 把描述嵌入当前消息传给 AI           │         │
│  │                                                │         │
│  │ 影响：当前这一条消息中的图片有没有描述         │         │
│  │ 只在插件决定处理当前消息时才会调用             │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  总结：                                                      │
│  • 平台的 → 负责让 "历史消息" 中的图片有描述               │
│  • 插件的 → 负责让 "当前消息" 中的图片有描述               │
│  • 两者互补，不冲突                                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 如果只开插件的图片处理，不开平台的？

历史消息中所有图片都会变成 `[Image]`（无描述），AI 在回复时看不懂历史中的图片内容。当前消息的图片可以处理，但上下文中的历史图片信息会丢失。

### 如果只开平台的，不开插件的？

历史消息中的图片会有描述，但当前触发回复的这条消息中的图片可能没有被处理（取决于插件的 `image_to_text_scope` 配置和概率筛选结果）。

### 推荐做法

**两个都开启**，各司其职：

| 设置 | 推荐值 | 说明 |
|------|--------|------|
| 平台 `image_caption`（自动理解图片） | **开启** | 确保历史消息中的图片有描述 |
| 平台 `image_caption_provider_id` | 配置一个轻量模型 | 用于自动描述历史图片 |
| 插件 `enable_image_processing` | **开启** | 确保当前消息中的图片有描述 |
| 插件 `image_to_text_provider_id` | 配置一个视觉模型 | 用于描述当前消息的图片 |

> 两者可以使用同一个 AI 提供商，也可以分别配置不同的模型（比如平台用便宜的，插件用精确的）。

## 推荐的平台设置

综合以上分析，使用本插件时，AstrBot 平台的"聊天记忆增强"应该这样配置：

| 平台配置项 | 推荐值 | 原因 |
|-----------|--------|------|
| 群聊上下文感知 | **开启** | 插件需要平台把群聊消息写入 platform_message_history 数据库 |
| 最大消息数 (`group_message_max_cnt`) | `10` ~ `50` | 只影响平台内存缓存（本插件会删掉注入的内容），设小节省内存 |
| 自动理解图片 (`image_caption`) | **开启** | 让历史消息中的图片有文字描述 |
| 群聊图片转述模型 | 配置一个视觉模型 | 用于生成图片描述文本 |
| 主动回复 (`active_reply`) | **关闭** | 本插件有自己的主动对话系统，两者会冲突 |

## 记忆插件的兼容性：为什么要用适配过的记忆插件？

### 实际上不止两次 AI 调用，而且调用路径并不相同

前文为了便于理解，主要先讲了**普通回复流程**。但如果讨论“为什么最好使用适配过的记忆插件”，只看普通回复是不够的。

实际运行时，本插件里和“记忆能不能被 AI 看见”直接相关的，至少有下面 4 类 AI 调用：

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ 1. 读空气决策 AI（要不要回复）                                              │
│    调用方式：provider.text_chat()                                           │
│    特点：直连 provider，完全绕过 Hook 链                                     │
│                                                                              │
│ 2. 普通回复生成 AI（回复什么）                                              │
│    调用方式：event.request_llm()                                             │
│    特点：会经过 Hook 链，未适配插件只有这里最容易注入记忆                     │
│                                                                              │
│ 3. 主动对话预判断 AI（现在适不适合主动开口）                                │
│    调用方式：provider.text_chat()                                           │
│    特点：同样直连 provider，完全绕过 Hook 链                                 │
│                                                                              │
│ 4. 主动对话生成 AI（主动说什么）                                            │
│    调用方式：provider.text_chat() + 运行前手动触发一次 on_llm_request Hook   │
│    特点：虽然会尝试兼容其他插件，但没有“当前用户短消息”可供未适配记忆插件检索 │
└──────────────────────────────────────────────────────────────────────────────┘
```

所以，**“未适配的记忆插件能不能生效”不能只看普通回复最终生成这一步，而要看它能不能覆盖整条链路里的不同 AI 调用方式。**

### 未适配的记忆插件：通常只有普通回复流程的最终生成 AI 能稳定看到记忆

假设你安装了一个本插件**未适配**的记忆插件（不是 LivingMemory 也不是 Legacy），它通常依赖 AstrBot 的标准 `on_llm_request` Hook 工作，常见方式是：

- 在 Hook 里向 `system_prompt` 注入记忆内容；或
- 在 Hook 里拿 `req.prompt` 当检索词，再召回记忆后注入。

这种插件在不同 AI 调用点上的表现，其实是这样的：

```
场景：使用未适配的记忆插件
════════════════════════════════════════════════════════════

  1. 读空气决策 AI（provider.text_chat）
  ┌────────────────────────────────────────┐
  │  记忆：❌ 没有                         │
  │  原因：直接调用 provider，不经过 Hook │
  └────────────────────────────────────────┘

  2. 普通回复生成 AI（event.request_llm）
  ┌────────────────────────────────────────┐
  │  记忆：✅ 有机会看到                   │
  │  原因：会经过 Hook，且这里有“当前用户 │
  │  消息短文本”可供未适配记忆插件检索    │
  └────────────────────────────────────────┘

  3. 主动对话预判断 AI（provider.text_chat）
  ┌────────────────────────────────────────┐
  │  记忆：❌ 没有                         │
  │  原因：同样直接调用 provider，不经过  │
  │  Hook                                  │
  └────────────────────────────────────────┘

  4. 主动对话生成 AI（text_chat + 手动触发 Hook）
  ┌────────────────────────────────────────┐
  │  记忆：❌ 通常也看不到                 │
  │  原因：虽然流程里会尝试触发 Hook，     │
  │  但主动对话没有“当前用户这句话”这个    │
  │  检索入口；对依赖标准 Hook / req.prompt│
  │  的未适配记忆插件来说，通常拿不到有效  │
  │  检索词，因此无法稳定注入有意义记忆    │
  └────────────────────────────────────────┘
```

也就是说，很多人容易误以为“未适配的记忆插件也能用”，其实往往只是因为**普通回复最终生成**这一步看起来像是生效了。

但从整条链路看：

- **读空气 AI** 看不到记忆
- **主动对话预判断 AI** 看不到记忆
- **主动对话生成 AI** 对未适配插件来说通常也看不到稳定、有效的记忆
- 真正比较容易看到记忆的，往往只有 **普通回复流程的最终生成 AI**

**结论：未适配的记忆插件通常只能影响“普通回复最后说什么”，却无法稳定影响“要不要回复”“要不要主动开口”以及“主动对话说什么”这些更前面的关键判断和生成环节。**

### 适配过的记忆插件：关键流程都能按插件自己的方式拿到记忆

当使用本插件适配的记忆插件（LivingMemory 或 Legacy），并开启 `enable_memory_injection` 时，情况就不一样了：

- 本插件不会被动等待外部 Hook“碰巧注入成功”
- 而是会**主动调用记忆插件接口取回记忆文本**
- 然后把记忆嵌入到自己掌控的上下文里

因此，只要是本插件自己构建上下文的关键 AI 流程，适配插件都能跟着生效。

```
场景：使用适配过的记忆插件 + enable_memory_injection = true
════════════════════════════════════════════════════════════

  普通回复流程
  ┌─────────────────── memory_insertion_timing 配置 ───────────────────┐
  │                                                                     │
  │  "pre_decision"（决策前注入）        "post_decision"（决策后注入） │
  │                                                                     │
  │  读空气决策 AI        ✅ 有记忆      读空气决策 AI        ❌ 无记忆 │
  │  普通回复生成 AI      ✅ 有记忆      普通回复生成 AI      ✅ 有记忆 │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘

  主动对话流程
  ┌─────────────────────────────────────────────────────────────────────┐
  │  主动对话链路里，记忆会先被嵌入 final_message，随后再进入：         │
  │                                                                     │
  │  1. 主动对话预判断 AI   ✅ 有记忆                                   │
  │  2. 主动对话生成 AI     ✅ 有记忆                                   │
  │                                                                     │
  │  也就是说，主动对话这条链路里，预判断和最终生成都会看到同一份记忆。 │
  └─────────────────────────────────────────────────────────────────────┘
```

其中需要特别注意：

- `memory_insertion_timing` 主要影响的是**普通回复流程里的读空气决策 AI**是否能提前看到记忆
- **主动对话流程**里，记忆注入发生在预判断和生成之前，因此这两步都能看到记忆

### 完整对比

| 场景 | 读空气决策 AI | 普通回复生成 AI | 主动对话预判断 AI | 主动对话生成 AI | 说明 |
|------|:---:|:---:|:---:|:---:|------|
| 未适配的记忆插件 | ❌ | ✅ | ❌ | ❌ | 外部插件依赖标准 Hook；真正稳定生效的通常只有普通回复最终生成 |
| 适配插件 + `post_decision` | ❌ | ✅ | ✅ | ✅ | 普通回复在决策后注入；主动对话链路仍会在预判断/生成前拿到记忆 |
| 适配插件 + `pre_decision` | ✅ | ✅ | ✅ | ✅ | 普通回复与主动对话两条链路都能在关键步骤看到记忆 |

### 判断型AI为什么单独支持人格开关/指定人格？

本插件现在只给三类“判断型AI”提供独立的人格控制：

- 读空气AI
- 主动对话预判断AI
- 频率判断AI

原因不是它们更重要，而是它们的职责不同：

- 这三者的本质任务都是**做判断**
- 回复生成AI / 主动对话生成AI 的本质任务是**直接按当前会话人格输出内容**

当某个人格提示词写得很强、很像“必须沉浸扮演”的角色卡时：
- 对生成型AI来说，这通常是合理的
- 但对判断型AI来说，模型可能会把自己理解成“我现在要进入角色去表演”，而不是“我要先冷静判断该不该回 / 该不该主动开口 / 现在是不是话太多”

因此，判断型AI现在支持：
- 保持默认：跟随当前会话当前生效人格
- 单独关闭人格注入：改为中性判断任务
- 单独指定一个更适合做判断的人格

这是一种**最小侵入**方案：只动最容易被人格干扰的判断层，不改变最终回复生成层原本应有的会话人格行为。

### 回复生成AI为什么仍然跟随当前会话人格？

因为回复生成AI和主动对话生成AI的职责，本来就是直接产出要发出去的话：

- 它们需要真正“像这个人格一样说话”
- 它们不该脱离当前会话人格单独运行
- 它们仍然会在每次调用时重新获取当前会话当前生效的人格，因此切换人格后可以立即生效

### 判断型AI的人格选择规则

三类判断型AI统一遵循以下规则：

1. **默认**：自动包含人格，且默认取当前会话当前生效的人格
2. **若关闭包含人格**：按中性判断任务执行，不额外注入人格
3. **若填写指定人格名**：只对该判断型AI固定使用该人格
4. **若指定人格名无效**：自动回退到当前会话人格，不中断主流程

⚠️ 指定人格名必须填写**完整名称**，不会做模糊匹配。
### 为什么决策 AI 有没有记忆很重要？

决策 AI 的任务是判断"现在要不要回复这条消息"。在本插件的决策提示词中，明确写了：

> **有记忆时更倾向于回复**，特别是：
> - 消息与记忆内容高度相关
> - 记忆显示与当前发送者有重要互动历史

也就是说，如果决策 AI 有记忆，它能做出更"人性化"的判断：

```
没有记忆的决策 AI：
  "张三说了'还记得上次那件事吗？'" → 不知道什么事 → 可能判断为不回复

有记忆的决策 AI：
  "张三说了'还记得上次那件事吗？'"
  + 记忆显示：上次和张三讨论了一个有趣的话题
  → 这个人跟我聊过！这条消息很相关！→ 判断为应该回复
```

### 为什么外部记忆插件不能注入到决策 AI？

根本原因是**调用方式不同**：

```
决策 AI：
  provider.text_chat(prompt, contexts, system_prompt)
  │
  │  这是直接调用 AI 提供商的底层 API
  │  完全不触发平台的 Hook 机制
  │  外部插件无法感知这次调用的存在
  │
  └→ 所以外部记忆插件的 on_llm_request Hook 根本不会被触发

回复 AI：
  event.request_llm(prompt, contexts, system_prompt)
  │
  │  这是通过平台标准流程调用
  │  会触发完整的 Hook 链
  │  外部插件可以在 Hook 中修改请求
  │
  └→ 外部记忆插件的 Hook 会被触发，可以注入到 system_prompt
```

**这不是设计缺陷，而是有意为之**——决策 AI 需要轻量快速地做判断，如果走 Hook 链，其他插件的修改可能干扰决策逻辑。但这也意味着，只有本插件**主动适配**的记忆插件才能把记忆送到决策 AI 面前。

### 推荐做法

| 推荐等级 | 做法 | 效果 |
|:---:|------|------|
| ⭐⭐⭐ | 使用适配的记忆插件 + `pre_decision` | 决策和回复都有记忆，AI 表现最好 |
| ⭐⭐ | 使用适配的记忆插件 + `post_decision` | 回复有记忆，决策无记忆 |
| ⭐ | 使用未适配的记忆插件 | 只有回复有记忆，决策完全无记忆 |

**目前适配的记忆插件：**

| 插件 | 配置模式 | 说明 |
|------|----------|------|
| [strbot_plugin_play_sy](https://github.com/kjqwer/strbot_plugin_play_sy) | `memory_plugin_mode: "legacy"` | 传统模式，稳定性高 |
| [astrbot_plugin_livingmemory](https://github.com/lxfight-s-Astrbot-Plugins/astrbot_plugin_livingmemory) | `memory_plugin_mode: "livingmemory"` | 智能模式，混合检索+人格隔离 |
| 自动检测（推荐） | `memory_plugin_mode: "auto"` | 自动检测已安装的记忆插件，无需手动选择 |

### auto 模式详解

`memory_plugin_mode: "auto"`（默认值）会在每次需要记忆注入时**自动检测**当前已安装且可用的记忆插件，按以下优先级选择：

```
检测优先级：LivingMemory v2 → LivingMemory v1 → Legacy
```

| 安装情况 | auto 模式行为 |
|----------|---------------|
| 只安装了 LivingMemory | 自动使用 LivingMemory（自动检测 v2/v1 版本） |
| 只安装了 Legacy | 自动使用 Legacy |
| 两个都安装了 | **优先使用 LivingMemory**（功能更强大），忽略 Legacy |
| 两个都没安装 | **自动跳过记忆注入**，不会报错，不影响其他功能正常运行 |

> **为什么推荐 auto 模式？**
> - 新用户无需了解两种记忆插件的区别，安装任意一个即可自动适配
> - 切换记忆插件时无需修改配置，卸载旧的、安装新的即可
> - 未安装记忆插件时不会产生错误日志，等安装后自动生效

### LivingMemory 双层兼容说明

当 `memory_plugin_mode = "livingmemory"` 或 auto 检测到 LivingMemory 时，当前版本会同时处理两层兼容：

1. **插件架构兼容**：`livingmemory_version`
   - `auto`（推荐）：自动识别当前插件是 `v2 initializer` 架构还是 `v1` 直挂架构
   - `v2`：强制按新架构读取 `initializer.memory_engine`
   - `v1`：强制按旧架构读取 `memory_engine`
2. **人格接口兼容**：`livingmemory_persona_compat_mode`
   - `auto`（推荐）：依次尝试 `resolve_selected_persona` → `get_default_persona_v3` → `get_personas_by_key`
   - `resolver_only` / `legacy_only` / `off`：用于排查特殊环境

> Web 面板技术树的「记忆注入」节点也已经同步展示这两个配置项，可直接在可视化界面中切换并保存。

## 其他常见问题

### 开启工具提醒后，某些 Skill / 工具出现参数串扰怎么办？

如果你发现这类现象：

- 某个 Skill / MCP 工具在**关闭本插件**时正常，**开启本插件**后更容易报错；
- 报错内容类似：
  - `unexpected keyword argument 'silent'`
  - `Tool handler parameter mismatch`
  - 某个工具收到了明显属于**另一个工具**的参数；
- 常见于需要在同一轮里混合调用 shell / python / send_message 等多种工具的 Skill；

通常说明问题更像是**工具参数提示过细导致的参数串扰**，而不是工具本身已经损坏。

#### 已内置的兼容处理

本插件现在会根据 AstrBot 当前会话的 `provider_settings.tool_schema_mode` 自动调整工具提醒细节：

- `skills_like`：只展示**工具名称 + 功能描述**，不再提前展开参数列表；
- `full`：继续展示**名称 + 描述 + 参数**；
- 旧版 AstrBot 未提供该配置时：按 `full` 兼容处理。

这样做是为了尽量不干扰 AstrBot 在 `skills_like` 下原本的两阶段工具 schema 暴露与 re-query 流程。

#### 建议排查顺序

1. **先确认 AstrBot 当前会话的 `tool_schema_mode`**
   - 如果是 `skills_like`，现在工具提醒应当只显示名称/描述；
   - 如果你仍然看到了参数列表，说明当前会话没有正确走到新的提醒降级逻辑。

2. **对照报错工具的真实签名检查是否发生了“跨工具参数混用”**
   - 例如 `astrbot_execute_shell` 只接受 `command / background / env`；
   - 若日志里却出现了 `silent` 之类本应属于 Python 工具的参数，就属于典型串扰。

3. **确认问题是否只出现在少数 Skill / MCP 工具上**
   - 如果主要集中在“同一轮混合调用多种工具”的 Skill，基本符合参数串扰特征；
   - 如果所有工具都普遍失败，则更应检查平台侧工具注册或上游模型状态。

4. **临时验证手段**
   - 保持插件开启，先关闭 `enable_tools_reminder` 再复测；
   - 若问题明显缓解，通常可以确认是提醒层提示干扰，而不是实际工具集损坏。

#### 什么时候应继续排查平台而不是插件

如果报错表现为：

- `send_message_to_user` 发送超时；
- MCP 服务器连接失败；
- `Tool not found` / `Server unavailable`；
- 工具返回内容本身为空或上游接口报错；

那更可能是 AstrBot / MCP / 协议端 / 上游服务问题，而不是本插件的工具提醒文本导致。


### Q: 我用了其他"智能回复"插件，为什么 AI 行为异常？

A: 如果另一个插件也修改 `req.contexts` 或 `req.prompt`，两者会互相覆盖。由于本插件优先级最低（-1），它会覆盖其他插件对这两个字段的修改。**建议只使用一个上下文管理类插件。**

### Q: 为什么 AstrBot 的 `/reset` 清除历史后，AI 还记得之前的对话？

A: 因为本插件使用**自己的消息存储**（custom_storage），不依赖平台的 `conversations` 表。`/reset` 只清除平台的表，不影响插件存储。请使用插件的 `gcp_reset` 或 `gcp_reset_here` 指令。

### Q: 其他修改 `system_prompt` 的插件能和本插件一起用吗？

A: 可以。只要它是在 `system_prompt` 前面插入或后面追加内容（对应兼容条件①②），本插件的 SystemPromptRewriter 会自动识别并保留。即使识别失败也会进入保守回退模式，优先保留当前内容。

### Q: 开启 `enable_debug_log` 后日志中看到的上下文是什么？

A: 是插件最终发送给 AI 的完整内容，包括格式化的历史消息、系统指令、当前消息等。这是排查问题的最佳方式。

### Q: 不开启平台的"群聊上下文感知"会怎样？

A: 插件将无法从 `platform_message_history` 表中读取历史消息，会回退到自定义存储。自定义存储的数据可能不如平台存储完整（特别是在插件刚安装、或者重启后）。**强烈建议开启。**

### Q: `group_message_max_cnt` 设成 300 会不会浪费？

A: 会略微浪费内存，但不会影响功能。平台会在内存中维护 300 条消息的缓存，然后在 on_llm_request 中注入到 system_prompt，但本插件会立即删掉这部分注入的内容。设成 10-50 更节省资源。

### Q: 平台的"自动理解图片"和插件的"图片处理"会重复调用 API 吗？

A: 对于**历史消息**中的图片，只有平台会调用一次（存储时）。对于**当前正在处理的消息**中的图片，如果插件决定处理，会再调用一次。两者作用在不同时机，不会对同一张图片重复调用。

## AI 回复频率相关问题

### Q: AI 回复频率太高，一直在刷屏怎么办？

A: 回复频率过高通常是概率和限制相关配置偏高导致的。按优先级依次排查和调整：

| 排查项 | 配置项 | 建议操作 |
|--------|--------|----------|
| 1. 基础概率过高 | `initial_probability` | 降低到 `0.05` ~ `0.08`（默认 `0.08`），这是没有注意力加成时的基础触发概率 |
| 2. 连续对话概率过高 | `after_reply_probability` | 降低到 `0.5` ~ `0.7`（默认 `0.8`），这是关闭注意力机制时传统模式的回复后临时提升概率 |
| 3. 回复密度限制未开启 | `enable_reply_density_limit` | 确保为 `true`，限制短时间内的最大回复次数 |
| 4. 密度窗口/上限 | `reply_density_window_seconds` / `reply_density_max_replies` | 默认 300 秒内最多 4 次，可改为 `300` 秒 / `2~3` 次 |
| 5. 对话疲劳未开启 | `enable_conversation_fatigue` | 确保为 `true`，连续对话后自动降低回复意愿 |
| 6. 时段概率未配置 | `enable_dynamic_reply_probability` | 开启后配置 `reply_time_periods`，深夜/午休时段自动降低频率 |
| 7. 平台主动回复未关闭 | AstrBot 聊天记忆增强 → 主动回复 | **必须关闭**，否则平台和插件会同时回复，频率翻倍 |

**快速降频方案**：将 `initial_probability` 改为 `0.03`，`after_reply_probability` 改为 `0.5`，`reply_density_max_replies` 改为 `2`。

### Q: AI 回复频率太低，基本不说话怎么办？

A: 回复频率过低的排查方向：

| 排查项 | 配置项 | 建议操作 |
|--------|--------|----------|
| 1. 基础概率太低 | `initial_probability` | 提高到 `0.10` ~ `0.15` |
| 2. 是否处于深夜/午休低频时段 | `reply_time_periods` | 检查当前时间是否命中低频时段（`factor < 1.0`），可调大对应 `factor` |
| 3. 对话疲劳阈值过低 | `fatigue_threshold_light` | 提高轻度疲劳阈值，如改为 `5`，让 AI 能连续多聊几轮 |
| 4. 频率调整器持续降频 | `enable_frequency_adjuster` | 暂时关闭观察，或调高 `frequency_min_probability`（如 `0.05`） |
| 5. 决策 AI 总是回答 NO | `decision_ai_provider_id` | 确认决策模型正常工作，开启 `enable_debug_log` 查看决策结果 |
| 6. 群组未启用 | `enabled_groups` | 留空 = 全部启用；如果填了群号，确认目标群在列表中 |
| 7. 注意力机制降低了概率 | `attention_decreased_probability` | 当 AI 对某用户注意力很低时会使用此值，可适当提高；如果你在使用传统模式，请确认 `enable_attention_mechanism` 已关闭，此项不会生效 |

**快速提频方案**：将 `initial_probability` 改为 `0.15`，`after_reply_probability` 改为 `0.85`，暂时关闭 `enable_conversation_fatigue`。

### Q: AI 回复频率忽高忽低、不稳定？

A: 这通常是多个动态系统叠加的正常表现。如果波动过大，可以逐一排查：

1. **频率调整器**（`enable_frequency_adjuster`）：每隔一段时间自动分析群聊节奏并调整概率，可能导致突然升降。如果不需要可以关闭
2. **注意力机制**（`enable_attention_mechanism`）：对不同用户有不同注意力值，会影响触发概率；关闭后会完全回退到传统回复后提升模式，不会再触发注意力溢出、注意力冷却等注意力专属机制
3. **动态时段概率**（`enable_dynamic_reply_probability`）：在时段切换的过渡期（`reply_time_transition_minutes`）概率会平滑变化
4. **对话疲劳**（`enable_conversation_fatigue`）：连续对话后突然降频是疲劳系统生效，等 `fatigue_reset_threshold` 秒后会恢复

**建议**：开启 `enable_debug_log`，观察日志中各系统对概率的实际影响值。

## AI 读空气判断相关问题

### Q: AI 读空气总是判断为不回复（NO），该怎么调？

A: "读空气"由决策 AI 负责，它会综合上下文判断当前是否应该回复。如果总是 NO：

1. **检查决策 AI 模型**：`decision_ai_provider_id` 所配置的模型是否正常工作？某些轻量模型可能理解力不足，总是保守回答 NO
2. **调整决策提示词**：通过 `decision_ai_extra_prompt` 追加提示，例如：`"你应该更积极地参与对话，当有人在讨论有趣的话题时倾向于回复 YES"`
3. **检查提示词模式**：`decision_ai_prompt_mode` 设为 `"append"` 是追加模式，`"override"` 会完全替换内置提示词（不推荐，除非你很清楚在做什么）
4. **注意力和记忆**：如果用了适配的记忆插件且 `memory_insertion_timing` 设为 `"pre_decision"`，记忆会帮助决策 AI 做出更准确的判断。没有记忆时决策 AI 可能偏保守
5. **概率筛选先行**：读空气判断是第二层筛选，消息必须**先通过概率筛选**才会进入读空气判断。确保 `initial_probability` 不会太低导致大部分消息在概率筛选阶段就被过滤了

### Q: AI 读空气判断太宽松，什么消息都回复？

A: 如果 AI 几乎对所有消息都判断 YES：

1. **检查是否跳过了决策 AI**：@消息、触发关键词等会跳过概率和读空气判断直接回复，确认是否大量消息通过这些旁路触发
2. **调整决策提示词**：通过 `decision_ai_extra_prompt` 追加限制，例如：`"只有当消息明确与你相关、直接提问、或话题非常有趣时才回复 YES，闲聊和水群不需要回复"`
3. **更换决策模型**：某些模型可能倾向于积极回应，尝试换一个更"理性"的模型
4. **利用消息质量预判**：开启 `enable_message_quality_scoring`，纯水聊消息会被降权，减少到达决策 AI 的低质量消息

### Q: 怎么查看 AI 读空气的具体判断过程？

A: 开启 `enable_debug_log`，日志中会显示：
- 概率筛选的计算过程和结果
- 决策 AI 收到的上下文内容
- 决策 AI 的回答（YES / NO）及原因
- 各个系统（注意力、疲劳、时段等）对概率的调整详情

## Web 管理面板相关问题

### Q: Web 管理面板的默认端口是什么？怎么访问？

A: Web 管理面板的相关信息：

| 项目 | 说明 |
|------|------|
| **默认端口** | `1451` |
| **默认监听地址** | `0.0.0.0`（所有网络接口） |
| **访问地址（本机）** | `http://localhost:1451` 或 `http://127.0.0.1:1451` |
| **访问地址（远程）** | `http://你的服务器IP:1451` |
| **默认状态** | 关闭（需在插件配置中将 `enable_web_panel` 设为 `true`） |

**首次使用步骤**：

1. 在 AstrBot 插件配置中将 `enable_web_panel` 设为 `true`
2. 重启 AstrBot（或重载插件）
3. 查看 AstrBot 日志，首次启动会以 **WARNING 级别**输出**随机生成的默认密码及安全警告**
4. 浏览器访问 `http://你的IP:1451`，使用默认密码登录
5. **登录后请立即修改为自定义密码**（修改后日志中将不再输出任何密码信息，明文副本自动删除）

### Q: Web 面板打不开 / 无法访问怎么办？

A: 按以下步骤逐一排查：

**第一步：确认插件已启用 Web 面板**
- 检查 `enable_web_panel` 是否为 `true`
- 检查 AstrBot 日志中是否有 Web 面板启动成功的信息

**第二步：确认端口和地址**
- 确认 `web_panel_port`（默认 `1451`）没有被其他程序占用
- 如果修改过端口，确认访问的是正确端口

**第三步：检查防火墙**
- **Windows**：检查 Windows Defender 防火墙是否放行了对应端口
  ```
  # 以管理员身份运行 PowerShell，添加入站规则
  New-NetFirewallRule -DisplayName "GCP Web Panel" -Direction Inbound -Port 1451 -Protocol TCP -Action Allow
  ```
- **Linux**：检查 iptables / firewalld / ufw 是否放行
  ```bash
  # ufw
  sudo ufw allow 1451/tcp

  # firewalld
  sudo firewall-cmd --permanent --add-port=1451/tcp && sudo firewall-cmd --reload
  ```
- **云服务器**（阿里云/腾讯云/华为云等）：除了系统防火墙，还需在**云控制台的安全组**中放行对应端口

**第四步：Docker 部署额外检查**
- 确认容器启动时映射了端口：`-p 1451:1451`
- 如果使用 docker-compose，检查 `ports` 配置：
  ```yaml
  ports:
    - "1451:1451"
  ```
- 如果修改了 `web_panel_port`，映射也要对应修改：`-p 你的端口:你的端口`
- 确认容器内部监听地址为 `0.0.0.0`（默认值），而不是 `127.0.0.1`（`127.0.0.1` 在容器内只允许容器内部访问）

**第五步：反向代理检查**（如果使用 Nginx 等反代）
- 确认反代配置正确转发到 `127.0.0.1:1451`
- 如果反代与 Web 面板在**同一台机器**，即使未开启 `web_panel_trust_proxy`，系统也会因为检测到连接来源为回环地址（`127.0.0.1 / ::1`）而自动读取 `X-Real-IP / X-Forwarded-For` 获取真实 IP
- 如果反代**不在本机**，且后端看到的连接来源不是回环地址，则需要显式开启 `web_panel_trust_proxy` 才会信任代理头
- `web_panel_trust_proxy` 现已归类为**安全边界配置**：只能在 AstrBot 传统配置界面修改，Web 面板中仅只读展示

### Q: 忘记了 Web 面板密码怎么办？

A: 两种方式重置密码：

1. **通过配置重置**：在 AstrBot 插件配置中将 `web_panel_reset_password` 设为 `true`，然后重启。密码会被重置为新的随机值并以 WARNING 级别输出到日志中（含安全警告）。重置完成后此开关会自动关闭；新密码会直接按 `Argon2id` 哈希保存
2. **删除认证文件**：删除插件数据目录下的 `web_data/auth.json` 和 `web_data/jwt_secret.json` 文件，重启后会重新生成随机密码并输出到日志

### Q: Web 面板提示"IP 已被封禁"怎么办？

A: 多次输入错误密码会触发暴力破解防护（分级锁定）：

| 失败次数 | 锁定时长 |
|:--------:|:--------:|
| 5 次 | 30 秒 |
| 10 次 | 60 秒 |
| 15 次 | 5 分钟 |
| 20 次 | 10 分钟 |
| 30 次 | 30 分钟 |
| 50 次 | 1 小时 |
| >50 次（解锁后重试） | 封禁（默认永久，可配置） |

此外系统还会监控登录频率：若某 IP 在配置的窗口期内（默认 10 秒）失败达到阈值（默认 3 次），直接封禁 IP 跳过阶梯锁定。`web_panel_brute_force_window`（默认 3600 秒）控制在多长时间无活动后自动清零失败计数，避免偶尔输错密码的用户长期累积。以上所有参数均可通过配置项调整，详见 [配置项参考](CONFIG_REFERENCE.md#暴力破解防护)。

等待锁定时间过后即可重新尝试。如果确实忘记密码，请参考上面的密码重置方法。

### Q: `web_panel_brute_force_window` 设为 0 有什么风险？

A: 此配置项控制"失败计数重置窗口"。设为零意味着失败计数**永不自动清零**（除非重启插件或登录成功）。

**极端场景**：假设你某天手误输错密码 4 次（未触发锁定），之后 3 个月没再输错。某天你又输错 1 次 → 直接触发第 5 次锁定（30 秒）。再过几个月继续累积 → 触发更长锁定，最终可能触发永久封禁。

**为什么不修？** 这是有意设计：如果系统在关闭衰减时仍自动清理，慢速暴力破解攻击者可以故意拉长尝试间隔绕过阶梯锁定。对于需要最高安全级别的公网部署，`window=0` 保证攻击者每次尝试都被永久记录。

**建议**：
- 内网/本地部署：保持默认 `3600`（1 小时），平衡安全与易用
- 公网暴露但不想误伤：设为 `86400`（24 小时）或更大值
- 需要最高安全且接受误伤风险：设为 `0`，搭配 `web_panel_protected_ips` 保护管理员 IP

### Q: 如何下载当前配置文件？

A: 在「配置流程图」或「核心设置」页面右下角有一个半透明浮窗，显示当前配置文件名。浮窗右侧有一个 **下载箭头图标（↓）**，鼠标悬停时会提示「点击下载配置文件」，点击即可将当前插件配置文件下载到本地。

**安全说明**：
- 下载功能**只读**，不会修改、删除或覆盖服务器上的任何文件
- 前端**不会暴露服务器绝对路径**，下载请求由服务端内部确定文件位置，响应中仅包含文件名和文件内容
- 必须在**登录面板后**才能使用（通过完整 JWT + 会话验证 + IP 过滤 + 防爬虫 + 已登录速率限制链路）
- 未登录、令牌失效或会话过期的请求在中间件层即被拦截（返回 401），不会执行文件读取，不消耗服务器 I/O 资源
- 下载功能受 Web 面板所有安全机制统一管控：IP 黑白名单、封禁列表、暴力破解锁定、防爬虫自动封禁、已登录请求速率限制
- 下载的文件即为插件当前生效的配置文件（`astrbot_plugin_group_chat_plus_config.json`）
- 每次下载操作均会写入 **Web 面板访问日志**（在「访问日志」页面可直接查看），附注中会标明下载成功或失败原因

### Q: 下载的配置文件可以直接修改后上传覆盖吗？

A：**不可以。** 下载功能是**纯粹的只读操作**，只提供文件内容的单向输出。它不提供上传、覆盖、删除或任何写入能力。如果需要在 Web 端修改配置，请使用「配置流程图」页面的可视化配置编辑器或「核心设置」页面的配置项，修改后点击「保存并重启插件」即可。

### Q: 点击下载按钮后提示"下载失败"怎么办？

A: 按以下步骤逐一排查。**建议同时查看 Web 面板「访问日志」页面中对应请求的状态码和附注**，可快速定位问题。

**第一步：确认浮窗可见且按钮可用**
- 下载按钮仅当右下角配置文件浮窗**正常显示**时才能点击
- 如果配置面板（右侧滑出）已打开，浮窗会临时隐藏，此时按钮不可用；关闭配置面板后浮窗会自动恢复
- 切换至其他视图（如数据可视化、会话管理等）后浮窗也会隐藏，只有「配置流程图」和「核心设置」页面会显示

**第二步：根据错误弹窗中的 HTTP 状态码定位问题**
- **401** — 登录会话已过期或失效（密码被修改、服务端重启、IP 发生变化等），刷新页面重新登录即可
- **403** — 服务端拒绝了下载请求：可能是配置文件格式校验不通过、文件权限不足，或当前 IP 被封禁/未在白名单中。查看 AstrBot 日志和访问日志附注获取详细原因
- **404** — 配置文件尚未生成，请确认插件已完全启动并完成首次配置加载
- **500** — 服务端内部错误（读取异常、编码错误等），请查看 AstrBot 日志排查具体异常
- **网络错误（无状态码）** — fetch 请求本身失败，通常为网络连接问题

**第三步：检查配置文件是否存在**
- 下载功能读取的是插件真实配置文件。如果插件尚未完成首次初始化（刚安装、还未启动完全），配置文件可能尚未生成
- 等待插件完全启动后再试；如果问题持续，检查 AstrBot 日志中是否有配置文件相关的错误信息

**第四步：反向代理检查**（如果使用 Nginx/Caddy 等反代）
- 下载端点路径为 `/api/config/download`，确保反向代理**未拦截或阻断**此路径
- 该路径属于 Web 面板 API 的一部分，反代配置应将 `/api/` 路径完整转发到后端
- Nginx 示例（确认没有遗漏 `/api/` 路径的转发规则）：
  ```nginx
  location / {
      proxy_pass http://127.0.0.1:1451;
      proxy_set_header Host $host;
      proxy_set_header X-Real-IP $remote_addr;
      proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  }
  ```
  上述配置已覆盖所有路径（包括 `/api/config/download`），无需额外规则
- 如果反代配置中单独限制了 `/api/` 路径（如只放行了特定子路径），需要补充 `/api/config/download` 的放行规则
- 如果反代有 WAF 或访问控制规则，确认下载请求不会被误判为恶意行为而拦截

**第五步：网络与防火墙检查**
- 下载是由浏览器直接发起的 `fetch` 请求，与页面访问使用同一连接。如果能正常打开面板页面，网络层面通常没有问题
- 如果浏览器控制台显示网络错误（如 `NetworkError`），检查是否存在浏览器插件拦截了下载请求
- 若使用 Docker 部署，确认容器端口映射正确（`-p 1451:1451`）

**第六步：权限问题**
- 极少数情况下，服务端可能因文件系统权限问题无法读取配置文件
- 此时 AstrBot 日志中会输出 `下载配置文件失败：权限不足` 及具体路径，访问日志附注中也会记录
- 检查插件进程是否对配置文件有读取权限（通常只要插件能正常启动，就一定有读取权限）
